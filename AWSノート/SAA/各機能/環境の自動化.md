# 🛠 AWS環境の自動化（SAAレベル）

## 1. 自動化の目的

- **人手の作業を減らす** → ヒューマンエラー削減、効率化
    
- **一貫性のある環境構築** → 開発/テスト/本番の整合性を担保
    
- **スケーラビリティと運用効率** → 負荷やイベントに応じて自動拡張・縮小
    
- **IaC（Infrastructure as Code）** の実践
    

---

## 2. 代表的な自動化サービス

### 🔹 インフラ構築の自動化（IaC）

- **AWS CloudFormation**
    
    - JSON/YAMLテンプレートでリソースを一括作成・管理
        
    - 冪等性（同じテンプレートなら同じ環境を再現可能）
        
- **AWS CDK（Cloud Development Kit）**
    
    - Python, TypeScript, Javaなどプログラミング言語でIaCを記述可能
        

---

### 🔹 サーバー設定・構成管理

- **AWS Systems Manager (SSM)**
    
    - EC2やオンプレサーバーのパッチ適用、設定変更、自動化ドキュメント（SSM Document）
        
    - Session Managerで安全なリモートアクセス
        
- **OpsWorks**
    
    - ChefやPuppetベースの構成管理サービス
        

---

### 🔹 デプロイ / CI/CD

- **AWS CodeCommit** → Gitリポジトリ
    
- **AWS CodeBuild** → ビルド（テスト・パッケージ）
    
- **AWS CodeDeploy** → EC2やLambdaへのデプロイ自動化
    
- **AWS CodePipeline** → CI/CDパイプライン全体を自動化
    

---

### 🔹 スケーリング自動化

- **Auto Scaling Group (ASG)**
    
    - EC2をロードに応じて自動で増減
        
- **Elastic Load Balancing (ELB)**
    
    - トラフィック分散とヘルスチェック
        
- **Lambda（イベント駆動）**
    
    - サーバーレス実行、自動スケーリング
        

---

### 🔹 運用自動化

- **CloudWatch**
    
    - メトリクス・ログ収集
        
    - アラームに応じて自動アクション（例: Lambda起動）
        
- **CloudTrail**
    
    - API操作ログの収集、自動検知との連携（GuardDutyなど）
        

---

## 3. 試験でよく問われるシナリオ例

### シナリオ①：一貫した環境構築

> 複数のリージョンに同じ環境をデプロイしたい。どうする？  
> ✅ **CloudFormationスタック** を使う

---

### シナリオ②：EC2のパッチ適用

> 定期的にOSパッチを当てたい。どうする？  
> ✅ **Systems Manager Patch Manager** を利用

---

### シナリオ③：アプリのデプロイ自動化

> 新しいアプリバージョンを段階的にEC2へ反映したい。  
> ✅ **CodeDeploy（Blue/Greenデプロイ対応）** を利用

---

### シナリオ④：高トラフィックに対応

> アクセス急増時にEC2が落ちないようにしたい。  
> ✅ **Auto Scaling + ELB + CloudWatchアラーム**

---

### シナリオ⑤：開発から本番までの一貫したパイプライン

> コードコミットから自動的にビルド・テスト・デプロイしたい。  
> ✅ **CodePipeline + CodeBuild + CodeDeploy**

---

## 4. SAA試験対策ポイント

- **CloudFormationとSystems Managerの違い** を理解する
    
    - CFN → リソースそのものの作成・削除・変更
        
    - SSM → 既存インスタンスの運用・管理（パッチ/設定）
        
- **Code系サービスの役割分担** を区別する
    
    - Commit＝Git
        
    - Build＝ビルド
        
    - Deploy＝デプロイ
        
    - Pipeline＝一連の流れ
        
- **Auto Scaling** は「動的スケーリング」「スケジュールスケーリング」の両方を押さえる
    
- **Blue/Greenデプロイ** や **Canaryリリース** も出題されやすい
    

---

✅ まとめ  
AWS環境の自動化は、

- **CloudFormation / CDK** → インフラ自動化
    
- **Systems Manager** → 運用自動化
    
- **Code系サービス** → CI/CD自動化
    
- **Auto Scaling / ELB / Lambda** → スケーリング自動化  
    を理解してシナリオごとに使い分けるのがSAAのポイントです。
